# 러닝 챌린지 자동화 — 에이전트 빌드 가이드

> **이 문서는 사람이 정독하는 교재가 아니라, AI에게 통째로 던지는 "빌드 브리프"입니다.**
> 러닝 앱 스크린샷 한 장을 올리면 → AI가 거리·시간·페이스를 읽어 → 노션에 자동 기록하는 웹앱을, **AI와 함께 페이즈 단위로** 만듭니다.
> 만드는 사람이 코드를 직접 짜지 않습니다. **무엇을 만들지 결정 → AI에게 요청 → 결과 확인**만 합니다.

---

## 0. 이 문서 사용법

### 먼저, 본인 상황에 맞는 **분기**를 고르세요

| | **🤖 분기 A — 에이전틱 툴** | **💬 분기 B — 일반 LLM** |
|---|---|---|
| 해당 도구 | 클로드 코드·Cursor·Codex·Antigravity CLI 등 (터미널/IDE에서 **로컬 파일을 직접 만지는** 에이전트) | 브라우저 챗봇(ChatGPT·Claude·Gemini 등). **파일 접근 없음** |
| 이 문서를 주는 법 | 이 저장소를 **통째로** 에이전트에게 줌 (클론 또는 URL). 에이전트가 `lib/`·`api/`·`public/` **실제 코드를 정답지로 읽음** | 이 문서(특히 §1 스펙)를 **복사해 붙여넣음**. LLM은 코드만 보고 만듦 |
| 코드가 만들어지는 법 | 에이전트가 파일을 직접 생성·실행 | LLM이 코드를 뱉음 → **사람이 복사해 파일로 저장** |
| 이 문서의 무게중심 | 가벼움 — 페이즈 진행 + "내 노션에 맞춰 바꿀 것" 안내 | 무거움 — §1 스펙이 유일한 정보원 |

> 💡 유료 코딩 에이전트가 없어도 됩니다. 구글 **Antigravity CLI**가 개인 구글 계정으로 무료입니다(터미널에서 로컬 파일을 직접 만들어주는 에이전트 → **분기 A**). 공식: <https://antigravity.google>. 터미널 자체가 부담일 때만 분기 B를 쓰세요.

### 진행 원칙 (두 분기 공통)

- **한 번에 한 페이즈씩.** 페이즈를 마치면 다음으로 넘어가기 전에 §2의 "✅ 완료 기준"을 확인합니다.
- **페이즈가 끝날 때마다 핸드오프를 갱신합니다** (§3). 컨텍스트가 길어져 AI가 헤매기 시작하면, 새 세션을 열고 핸드오프부터 읽혀 이어가면 됩니다.
- **시크릿(토큰·키)은 절대 채팅창에 붙여넣거나 깃에 올리지 않습니다.** 항상 `.env.local` 파일과 배포 플랫폼 환경변수에만 둡니다.

### 💬 분기 B 전용 — 세션 시작 때 **이 블록을 한 번 붙여넣으세요** (LLM에 역할 부여)

> 너는 코딩을 처음 하는 사람을 1:1로 안내하는 빌드 가이드야. 아래 규칙을 끝까지 지켜:
> 1. **매 단계마다, 코드를 주기 전에 먼저** ① 지금 무엇을 ② 왜 하는지 ③ 끝나면 화면에서 뭘 보게 되는지를 **비전문가도 알아듣는 쉬운 말**로 설명해.
> 2. 그다음 코드를 주고, **"이 내용을 `○○` 파일로 저장하세요"** 처럼 어디에 둘지 정확히 알려줘.
> 3. **한 번에 한 단계씩.** 내가 "됐어" / "다음"이라고 하기 전에는 다음 단계로 넘어가지 마.
> 4. 단계가 끝나면 "이게 되는지" 확인하는 **점검 질문**을 주고, 막히면 어디를 볼지 알려줘.
> 5. **토큰·API 키 같은 비밀값은 절대 이 채팅에 붙여넣지 말라**고 매 단계 상기시켜. 그건 `.env.local` 파일에만 들어가야 해.
>
> 준비됐으면 "준비 완료"라고만 답하고, 내가 첫 페이즈를 줄 때까지 기다려.

> 🤖 분기 A는 위 역할 부여가 필요 없지만, 원하면 "작업하면서 지금 뭘 하는지 한국어로 짧게 설명하며 진행해줘"라고 한마디 덧붙이세요.

## 1. 프로젝트 스펙 (분기 공통 · 분기 B는 이 절이 핵심)

### 무엇을 만드나 — 자동화 1: 입력

```
[브라우저] 사진 업로드 화면
   │ ① 페이지 열면 챌린저 목록 요청
   │  ──▶ /api/challengers  ──▶ 노션 '챌린저' DB 읽기
   │ ② 본인 선택 + 러닝 앱 스크린샷 업로드
   │  ──▶ /api/upload
   │         ├─ lib/gemini.js : 사진 → 러닝 데이터(JSON)
   │         └─ lib/notion.js : 노션 '러닝 기록' DB에 새 페이지 + 사진 첨부
   │ ◀── 결과 카드(추출값 + 노션 링크)
```

> 핵심 한 줄: **AI가 닿는 곳은 "사진 읽기" 단 한 군데.** 목록 읽기·페이지 생성·사진 첨부·관계 연결은 전부 코드가 정해진 대로 처리합니다.

### 기술 스택

| 역할 | 기술 |
|------|------|
| 호스팅 / 서버리스 함수 | **Vercel** (`api/*.js` 파일 하나 = 엔드포인트 하나) |
| 이미지 인식(OCR) | Google **Gemini** Vision (`gemini-2.5-flash`) |
| 데이터 저장 | **노션** REST API (버전 `2026-03-11`, **`data_source_id`** 기반) |
| 런타임 / 프런트 | Node.js(ESM), 의존성 최소 · 바닐라 HTML/CSS/JS (`public/index.html`) |

### 노션 데이터 모델 — DB 2개를 미리 만들어 둡니다

노션 표(데이터베이스)의 한 행 = 한 "페이지". 칸 이름은 **글자·괄호·단위까지 아래와 똑같이** 만드세요 (특히 분기 B에서 이름이 다르면 기록이 빈 칸으로 들어갑니다).

**① 러닝 기록 DB** — 자동으로 채워지는 곳

| 칸 이름 | 유형 | 누가 채우나 |
|---------|------|-------------|
| `제목` | 제목(title) | 코드 (그날 날짜) |
| `일시` | 날짜(date) | 코드 |
| `챌린저` | 관계(relation) → 챌린저 DB | 코드 (누가 올렸는지 연결) |
| `인증샷` | 파일(files) | 코드 (업로드한 사진 첨부) |
| `거리(km)` `시간(초)` `평균 페이스(초/km)` `평균 심박수` `고도 상승(m)` `케이던스` `칼로리` | 숫자(number) | **Gemini가 읽은 값** |
| `시간(표시)` `평균 페이스(표시)` | 수식(formula) | 노션 자동 (초 → "32:15" 보기 좋게) |

**② 챌린저 DB** — 참가자 명단

| 칸 이름 | 유형 | 쓰임 |
|---------|------|------|
| `이름` | 제목(title) | 화면 드롭다운에 표시 |
| (페이지 아이콘) | 아이콘 | 화면에 동그란 아바타로 표시 |

> **관계(relation)가 핵심.** '러닝 기록'의 `챌린저` 칸을 '챌린저' DB와 연결해 두면, 누가 무엇을 올렸는지 노션이 자동으로 묶어줍니다.

### Gemini OCR 규칙 (lib/gemini.js의 프롬프트가 지켜야 할 것)

- **입력**: 제각각인 러닝 앱 스크린샷 (애플워치·가민·삼성헬스·NRC 등).
- **출력**: 프로그램이 바로 쓰도록 **정해진 칸만, JSON 한 덩어리**로.
  ```
  { 거리(km), 시간(초), 페이스(초/km), 심박수, 고도(m), 케이던스, 칼로리, 날짜, 앱이름 }
  ```
- **추측 금지**: 화면에 안 보이는 값은 지어내지 말고 **비워둠(null)**.
- **단위 정규화**: 마일 → km, `"32:15"` 같은 표시 → **초(1935)**.

### 시크릿 / 환경변수

- 모든 시크릿은 로컬 **`.env.local`** (→ `.gitignore`에 등록, 깃에 안 올라감) + 배포 시 **Vercel 환경변수**에만.
- **자동화 1(빌드 범위)에 필요한 변수:**

  | 변수 | 무엇 | 발급처 |
  |------|------|--------|
  | `NOTION_TOKEN` | 노션 인테그레이션 토큰 (`ntn_...`) | <https://www.notion.so/my-integrations> |
  | `NOTION_DS_RUNNING` | '러닝 기록' DB의 데이터소스 ID | 노션 GUI(아래) |
  | `NOTION_DS_CHALLENGERS` | '챌린저' DB의 데이터소스 ID | 노션 GUI(아래) |
  | `GEMINI_API_KEY` | Gemini API 키 | <https://aistudio.google.com/apikey> |
  | `NOTION_DASHBOARD_URL` | (선택) 결과 카드가 가리킬 공개 대시보드 URL | 노션 |

  > **데이터소스 ID 얻는 법(노션 GUI)**: DB 우상단 `⋯` → **데이터 소스 관리** → 출처 → 미트볼 `⋯` → **데이터 소스 ID 복사**. (2026-03-11 API는 `database_id`가 아니라 이 `data_source_id`를 씁니다.)
  > **인테그레이션 권한**: 위 토큰을 만든 인테그레이션을 DB 2개 각각의 `⋯ → 연결 추가`로 **초대**해야 합니다. 빼먹으면 401/권한 오류.

### 설계 원칙 — AI는 "읽기"만, 나머지는 코드

- **AI(Gemini)**: 사진 → 데이터 추출. 이 한 곳만.
- **코드(결정적)**: 누가 올렸는지 매칭, 빈 값 분기, 단위 정리, 노션 API 호출, 파일 첨부.
- 이유: 분기·반복·쓰기 같은 정해진 일을 매번 LLM에 묻는 건 느리고·비싸고·답이 흔들립니다. 정해진 일은 코드가 **즉시·일관·거의 무료**로.

### 범위 밖 — 자동화 2: 문자 발송 (개념만)

기록이 올라오는 순간(웹훅) 또는 매일 아침(크론) 챌린저에게 SMS를 보내는 기능이 저장소에 있습니다(`api/webhook/`, `api/cron/`, `lib/solapi.js`). **이 가이드의 빌드 범위가 아닙니다** — 유료 문자 서비스(Solapi) 계정·발신번호 등록이 필요합니다. 입력 자동화를 다 만든 뒤 같은 방식으로 확장하세요.

> 🤖 분기 A 주의: 저장소의 `scripts/`와 `sync-emoji-icons.mjs`는 작성자가 DB를 코드로 세팅한 **1회성 유틸이라 그대로는 실행되지 않습니다.** 정답지로 삼을 곳은 `lib/`·`api/`·`public/`입니다.

### 페이즈 0 · 셋업

🎯 **목표**: 멍석 깔기 — 계정·키·노션 DB 2개·인테그레이션·작업 폴더.

준비:

- 도구: Node.js. (분기 A는 코딩 에이전트도.)

💬 **분기 B**: 빈 작업 폴더를 만들고 — "러닝 챌린지 자동화를 만들 거야. 먼저 `.env.local`과 `.gitignore` 파일을 어떻게 만들고 무엇을 적는지 안내해줘. 비밀값은 내가 직접 넣을게."

✅ **완료 기준**: 노션 DB 2개 + 챌린저 몇 명, 손에 토큰·Gemini 키, 데이터소스 ID 2개, `.env.local` 준비.
📝 **핸드오프**: 분기(A/B), 작업 폴더 위치, DB 2개의 데이터소스 ID, `.env.local`에 채운 **키 이름들**(값은 적지 않음).

### 페이즈 2 · 사진 읽는 두뇌 (Gemini OCR)

🎯 **목표**: 러닝 앱 스크린샷 → 러닝 데이터(JSON). §1 "Gemini OCR 규칙" 그대로.

먼저 테스트용 스크린샷 몇 장을 폴더의 `samples/`에 넣습니다(저장소 `samples/`를 써도 됨).

🤖 **분기 A**: "저장소의 `lib/gemini.js`를 정답지로, 같은 OCR 규칙(추측 금지·마일→km·시간→초·정해진 JSON 칸)으로 내 `GEMINI_API_KEY`(`gemini-2.5-flash`)를 써서 만들어줘. 그리고 `samples/` 사진 한 장으로 추출 결과를 보여줘."
💬 **분기 B**: "러닝 앱 스크린샷에서 §스펙의 JSON 칸(거리(km)·시간(초)·페이스(초/km)·심박수·고도·케이던스·칼로리·날짜·앱이름)을 뽑는 코드를 Gemini(`gemini-2.5-flash`)로 만들어줘. **화면에 없는 값은 추측 말고 null**, 마일은 km로, 시간 표시는 초로 바꿔. 키는 `.env.local`의 `GEMINI_API_KEY`. 만들고 나면 샘플 사진 한 장으로 테스트하는 법도 알려줘."

✅ **완료 기준**: 샘플 사진에서 거리·시간·페이스가 **숫자로** 나오고, 화면에 없던 항목은 **비어 있음(null)**.

## 3. 핸드오프 프로토콜 (컨텍스트 한계 대비)

AI는 대화가 길어지면 앞부분을 잊거나 헤맵니다. 그래서 **페이즈마다 핸드오프를 남기고, 필요하면 새 세션에서 이어갑니다.**

### 페이즈를 마칠 때마다 — 작업 폴더에 `HANDOFF.md` 갱신

AI에게 "이번 페이즈 핸드오프를 `HANDOFF.md`에 갱신해줘"라고 요청하세요. 담을 내용:

```markdown
# HANDOFF — 러닝 챌린지 빌드

- 분기: A(에이전틱) / B(일반 LLM)
- 작업 폴더: <경로 또는 저장소>
- 완료한 페이즈: 0,1,2 ... / 현재: 페이즈 3 진행 중
- 현재 상태: <무엇까지 됐는지 한두 줄>
- 노션: 러닝 기록 DS=<id> · 챌린저 DS=<id>
- .env.local에 채운 키: NOTION_TOKEN, NOTION_DS_RUNNING, ... (값은 적지 않음)
- 다음 할 일: <다음 페이즈/구체 작업>
- 미결정·막힌 점: <있으면>
```

> ⚠️ `HANDOFF.md`에 **토큰·키 값은 절대 적지 마세요.** "어떤 키를 채웠다"는 이름만.

### 컨텍스트가 차면 — 새 세션에서 이어가기

1. 새 대화(세션)를 엽니다.
2. **분기 A**: 저장소 + `HANDOFF.md`를 에이전트에게 주고 → "`HANDOFF.md` 읽고 다음 페이즈부터 이어가줘."
   **분기 B**: ① 이 가이드(§0 역할 부여 + §1 스펙) → ② `HANDOFF.md` 내용 → 순서로 붙여넣고 → "여기부터 이어가자."
3. AI가 현재 상태를 요약하게 한 뒤, 다음 페이즈를 진행합니다.

---

막히는 개념·정확한 칸 이름·env 변수명이 헷갈리면 → **§1 스펙**으로 돌아오세요. 정답 코드가 궁금하면(분기 A) → 저장소의 `lib/`·`api/`·`public/`.
