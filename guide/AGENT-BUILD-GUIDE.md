# 러닝 챌린지 자동화 — 에이전트 빌드 가이드

> **이 문서는 사람이 정독하는 교재가 아니라, AI에게 통째로 던지는 "빌드 브리프"입니다.**
> 러닝 앱 스크린샷 한 장을 올리면 → AI가 거리·시간·페이스를 읽어 → 노션에 자동 기록하는 웹앱을, **AI와 함께 페이즈 단위로** 만듭니다.
> 만드는 사람이 코드를 직접 짜지 않습니다. **무엇을 만들지 결정 → AI에게 요청 → 결과 확인**만 합니다.

---

## 0. 이 문서 사용법

### 먼저, 본인 상황에 맞는 **분기**를 고르세요

| | **🤖 분기 A — 에이전틱 툴** | **💬 분기 B — 일반 LLM** |
|---|---|---|
| 해당 도구 | 클로드 코드·Cursor·Codex·Antigravity CLI 등 (터미널/IDE에서 **로컬 파일을 직접 만지는** 에이전트) | 브라우저 챗봇(ChatGPT·Claude·Gemini 등). **파일 접근 없음** |
| 이 문서를 주는 법 | 이 저장소를 **통째로** 에이전트에게 줌 (클론 또는 URL). 에이전트가 `lib/`·`api/`·`public/` **실제 코드를 정답지로 읽음** | 이 문서를 **복사해 붙여넣음**. LLM은 §1 스펙만 보고 만듦 |
| 코드가 만들어지는 법 | 에이전트가 파일 생성 → GitHub 푸시 | LLM이 코드를 주면 → 사람이 **GitHub 웹에서 파일로 만듦** |
| 이 문서의 무게중심 | 가벼움 — 페이즈 진행 + "내 노션에 맞춰 바꿀 것" 안내 | 무거움 — §1 스펙이 유일한 정보원 |

> 💡 유료 코딩 에이전트가 없어도 됩니다. 구글 **Antigravity CLI**가 개인 구글 계정으로 무료입니다(로컬 파일을 직접 만들어주는 에이전트 → **분기 A**). 공식: <https://antigravity.google>. 비코더라면 오히려 분기 A가 더 쉬울 수 있어요(에이전트가 파일 생성·배포를 대신 해줌). 브라우저만 쓰고 싶으면 분기 B로.

### 진행 원칙 (두 분기 공통)

- **내 컴퓨터에서 코드를 실행하지 않습니다.** 만든 파일은 GitHub에 올리고 **Vercel이 실행**합니다 — Node 설치·터미널·로컬 서버 불필요. 동작 확인은 **배포된 주소**에서.
- **한 번에 한 페이즈씩.** 페이즈를 마치면 §2의 "✅ 완료 기준"을 확인하고 넘어갑니다.
- **페이즈가 끝날 때마다 핸드오프를 갱신합니다** (§3). 컨텍스트가 길어져 AI가 헤매면, 새 세션에서 핸드오프부터 읽혀 이어갑니다.
- **시크릿(토큰·키)은 채팅·코드·GitHub 어디에도 노출하지 않습니다.** 오직 **Vercel 환경변수 화면**에만 입력합니다.

### 💬 분기 B 전용 — LLM 행동 규칙 (이 파일을 통째로 붙여넣으면 이 규칙대로 진행됩니다)

> 너는 코딩을 한 번도 안 해본 사람을 1:1로 안내하는 빌드 가이드야. 아래를 끝까지 지켜:
> 1. **모든 작업은 웹 브라우저에서만** 이뤄져 — 노션(웹), GitHub(웹에서 파일 만들고 올리기), Vercel(웹에서 배포). **프로그램 설치·터미널·메모장·VS Code는 전혀 필요 없어.** 전문 용어는 늘 쉬운 말로 풀어줘.
> 2. **진행 방식**: 아래 §2 페이즈를 **0번부터 순서대로 네가 직접 이끌어.** 한 페이즈 안에 여러 단계(예: 파일 여러 개 만들기)가 있으면 **한 단계씩** 진행하고, 페이즈 전체가 끝나면 내가 **"다음"**이라고 할 때까지 기다려.
> 3. 매 단계, 코드나 내용을 주기 **전에 먼저** ① 지금 뭘 ② 왜 ③ 끝나면 뭘 보게 되는지를 쉬운 말로 설명해.
> 4. 코드를 줄 때는 **GitHub에서 파일 만드는 법**을 정확히 안내해: "GitHub 저장소에서 **Add file → Create new file**, 위 이름칸에 `lib/gemini.js`처럼 경로째 입력, 아래에 이 코드를 붙여넣고 **Commit**." (이름에 `/`를 넣으면 폴더가 자동 생성돼.)
> 5. 한 단계 끝나면 "이게 됐는지" 확인하는 질문을 주고, 막히면 어디를 볼지 알려줘.
> 6. 토큰·API 키 같은 **비밀값은 절대 이 채팅에 붙여넣지 말라**고 매번 상기시켜 — 그건 나중에 **Vercel 환경변수 화면**에만 입력해.
>
> 이 규칙을 읽었으면 **인사나 요약 없이 곧바로 페이즈 0부터 시작해.**

> 🤖 분기 A는 위 역할 부여가 필요 없습니다. 에이전트에게 저장소를 주고 "이 가이드대로 페이즈 0부터, 로컬 실행 말고 GitHub→Vercel로 배포하며 진행해줘. 작업하면서 뭘 하는지 한국어로 짧게 설명해줘"라고 하면 됩니다.

## 1. 프로젝트 스펙 (분기 공통 · 분기 B는 이 절이 핵심)

### 무엇을 만드나 — 자동화 1: 입력

```
[브라우저] 사진 업로드 화면
   │ ① 페이지 열면 챌린저 목록 요청
   │  ──▶ /api/challengers  ──▶ 노션 '챌린저' DB 읽기
   │ ② 본인 선택 + 러닝 앱 스크린샷 업로드
   │  ──▶ /api/upload
   │         ├─ lib/gemini.js : 사진 → 러닝 데이터(JSON)
   │         └─ lib/notion.js : 노션 '러닝 기록' DB에 새 페이지 + 사진 첨부
   │ ◀── 결과 카드(추출값 + 노션 링크)
```

> 핵심 한 줄: **AI가 닿는 곳은 "사진 읽기" 단 한 군데.** 목록 읽기·페이지 생성·사진 첨부·관계 연결은 전부 코드가 정해진 대로 처리합니다.

### 기술 스택

| 역할 | 기술 |
|------|------|
| 호스팅 / 서버리스 함수 | **Vercel** (`api/*.js` 파일 하나 = 엔드포인트 하나, 푸시하면 자동 빌드·배포) |
| 이미지 인식(OCR) | Google **Gemini** Vision (`gemini-2.5-flash`) |
| 데이터 저장 | **노션** REST API (버전 `2026-03-11`, **`data_source_id`** 기반) |
| 런타임 / 프런트 | Node.js(ESM), 의존성 최소 · 바닐라 HTML/CSS/JS (`public/index.html`) |

### 노션 데이터 모델 — DB 2개를 미리 만들어 둡니다

노션 표(데이터베이스)의 한 행 = 한 "페이지". 칸 이름은 **글자·괄호·단위까지 아래와 똑같이** 만드세요 (이름이 다르면 기록이 빈 칸으로 들어갑니다).

**① 러닝 기록 DB** — 자동으로 채워지는 곳

| 칸 이름 | 유형 | 누가 채우나 |
|---------|------|-------------|
| `제목` | 제목(title) | 코드 (그날 날짜) |
| `일시` | 날짜(date) | 코드 |
| `챌린저` | 관계(relation) → 챌린저 DB | 코드 (누가 올렸는지 연결) |
| `인증샷` | 파일(files) | 코드 (업로드한 사진 첨부) |
| `거리(km)` `시간(초)` `평균 페이스(초/km)` `평균 심박수` `고도 상승(m)` `케이던스` `칼로리` | 숫자(number) | **Gemini가 읽은 값** |
| `시간(표시)` `평균 페이스(표시)` | 수식(formula) | 노션 자동 (초 → "32:15" 보기 좋게) |

**② 챌린저 DB** — 참가자 명단

| 칸 이름 | 유형 | 쓰임 |
|---------|------|------|
| `이름` | 제목(title) | 화면 드롭다운에 표시 |
| (페이지 아이콘) | 아이콘 | 화면에 동그란 아바타로 표시 |

> **관계(relation)가 핵심.** '러닝 기록'의 `챌린저` 칸을 '챌린저' DB와 연결해 두면, 누가 무엇을 올렸는지 노션이 자동으로 묶어줍니다.

### Gemini OCR 규칙 (lib/gemini.js의 프롬프트가 지켜야 할 것)

- **입력**: 제각각인 러닝 앱 스크린샷 (애플워치·가민·삼성헬스·NRC 등).
- **출력**: 프로그램이 바로 쓰도록 **정해진 칸만, JSON 한 덩어리**로.
  ```
  { 거리(km), 시간(초), 페이스(초/km), 심박수, 고도(m), 케이던스, 칼로리, 날짜, 앱이름 }
  ```
- **추측 금지**: 화면에 안 보이는 값은 지어내지 말고 **비워둠(null)**.
- **단위 정규화**: 마일 → km, `"32:15"` 같은 표시 → **초(1935)**.

### 시크릿 / 환경변수

- 비밀값(토큰·키)은 **Vercel 환경변수**(배포 화면)에만 등록합니다. 채팅·코드·GitHub엔 절대 노출 금지. (이 가이드는 로컬 실행이 없어 `.env.local` 파일을 쓰지 않습니다.)
- **자동화 1에 필요한 변수 4종:**

  | 변수 | 무엇 | 발급처 |
  |------|------|--------|
  | `NOTION_TOKEN` | 노션 인테그레이션 토큰 (`ntn_...`) | <https://www.notion.so/my-integrations> |
  | `NOTION_DS_RUNNING` | '러닝 기록' DB의 데이터소스 ID | 노션 GUI(아래) |
  | `NOTION_DS_CHALLENGERS` | '챌린저' DB의 데이터소스 ID | 노션 GUI(아래) |
  | `GEMINI_API_KEY` | Gemini API 키 | <https://aistudio.google.com/apikey> |

  > **데이터소스 ID 얻는 법(노션 GUI)**: DB 우상단 `⋯` → **데이터 소스 관리** → 출처 → 미트볼 `⋯` → **데이터 소스 ID 복사**. (2026-03-11 API는 `database_id`가 아니라 이 `data_source_id`를 씁니다.)
  > **인테그레이션 권한**: 위 토큰을 만든 인테그레이션을 DB 2개 각각의 `⋯ → 연결 추가`로 **초대**해야 합니다. 빼먹으면 권한 오류.

### 설계 원칙 — AI는 "읽기"만, 나머지는 코드

- **AI(Gemini)**: 사진 → 데이터 추출. 이 한 곳만.
- **코드(결정적)**: 누가 올렸는지 매칭, 빈 값 분기, 단위 정리, 노션 API 호출, 파일 첨부.
- 이유: 분기·반복·쓰기 같은 정해진 일을 매번 LLM에 묻는 건 느리고·비싸고·답이 흔들립니다. 정해진 일은 코드가 **즉시·일관·거의 무료**로.

### 범위 밖 — 자동화 2: 문자 발송 (개념만)

기록이 올라오는 순간(웹훅) 또는 매일 아침(크론) 챌린저에게 SMS를 보내는 기능이 저장소에 있습니다(`api/webhook/`, `api/cron/`, `lib/solapi.js`). **이 가이드의 빌드 범위가 아닙니다** — 유료 문자 서비스(Solapi) 계정·발신번호 등록이 필요합니다. 입력 자동화를 다 만든 뒤 같은 방식으로 확장하세요.

> 🤖 분기 A 주의: 저장소의 `scripts/`와 `sync-emoji-icons.mjs`는 작성자가 DB를 코드로 세팅한 **1회성 유틸이라 그대로는 실행되지 않습니다.** 정답지로 삼을 곳은 `lib/`·`api/`·`public/`입니다.

## 2. 페이즈 (0 → 3)

> 원칙: **파일은 만들되 실행은 Vercel에서.** 내 컴퓨터에서 코드를 돌리지 않습니다(설치·터미널 불필요). 조각조각 확인하지 않고, **다 만들어 배포한 뒤 라이브 주소에서 한 번에** 검증합니다.

각 페이즈 형식: **🎯 목표 / 🤖 분기 A / 💬 분기 B / ✅ 완료 기준 / 📝 핸드오프**

> 💬 분기 B는 LLM이 §0 규칙대로 페이즈 0부터 순서대로 직접 이끕니다. 사용자는 한 페이즈가 끝날 때 **"다음"**이라고만 하면 됩니다.

### 페이즈 0 · 셋업 (노션·계정·키)

🎯 **목표**: 노션 DB 2개와 열쇠(키)를 준비합니다. **코드는 아직 안 만듭니다.**

- 계정 4개: 노션 · Google AI Studio · GitHub · Vercel (전부 무료 시작, Vercel은 GitHub 계정으로 로그인 가능).
- §1 데이터 모델대로 노션 **DB 2개** 생성 → '챌린저' DB에 참가자 몇 명 추가('러닝 기록'은 빈 채로).
- 노션 인테그레이션 토큰 발급 → DB 2개에 **연결 추가**(초대). 데이터소스 ID 2개 복사(§1 GUI 방법).
- 키 4종을 잠깐 메모해 둠(곧 Vercel에 입력): `NOTION_TOKEN`·`NOTION_DS_RUNNING`·`NOTION_DS_CHALLENGERS`·`GEMINI_API_KEY`. **채팅엔 붙여넣지 않습니다.**

🤖 **분기 A**: "이 저장소 구조와 `lib/`·`api/`·`public/`을 파악해줘. 곧 내 노션 DB(데이터소스 ID 2개)에 맞춰 만들 거야."
💬 **분기 B**: LLM과 함께 노션 DB 2개를 만들고 키 4종을 모았는지 점검. (파일은 아직 안 만듦)

✅ **완료 기준**: 노션 DB 2개 + 챌린저 몇 명, 데이터소스 ID 2개, 키 4종 확보.
📝 **핸드오프**: 분기(A/B), DB 2개의 데이터소스 ID, 확보한 키 **이름들**(값 X).

### 페이즈 1 · 코드 만들어 GitHub에 올리기

🎯 **목표**: 앱을 이루는 파일들을 만들어 **비공개 GitHub 저장소**에 넣습니다(Vercel이 여기서 받아감). **로컬 실행은 안 합니다.**

만들 파일:
- `lib/gemini.js` — 사진 → JSON (§1 OCR 규칙)
- `lib/notion.js` — 노션 읽기/쓰기 (`data_source_id` 기반)
- `api/challengers.js` — 챌린저 목록 주는 입구
- `api/upload.js` — 사진 받아 OCR→노션 기록 (핵심)
- `api/debug.js` — 키 등록 점검용 (값은 안 보임 · 페이즈 3 검증에 씀)
- `public/index.html` — 본인 선택 + 사진 업로드 화면
- `package.json` — 코드가 쓰는 부품 목록 (Vercel이 이걸 보고 자동 설치)

🤖 **분기 A**: "저장소의 `lib/`·`api/`·`public/`을 정답지로, 같은 구성의 파일들을 내 노션 DB(데이터소스 ID 2개)에 맞춰 만들고 **비공개 GitHub 저장소로 push**해줘. 로컬에서 실행하진 말고 — Vercel로 배포할 거야. (OCR은 §1 규칙: 추측금지·마일→km·시간→초·정해진 JSON)"
💬 **분기 B**:
  1. github.com → New repository → **Private**로 빈 저장소 생성.
  2. LLM이 위 파일을 **하나씩** 코드와 함께 줍니다. 각 파일을 저장소에서 **Add file → Create new file**로 만듭니다(이름칸에 `lib/gemini.js`처럼 경로째 입력 → 코드 붙여넣기 → Commit). 폴더는 자동 생성됩니다.
  (LLM이 파일마다 안내. 비밀 키는 코드에 안 넣으니 올라갈 위험 없음.)

✅ **완료 기준**: GitHub 저장소에 위 파일들이 보입니다. (동작 확인은 배포 후 페이즈 3에서.)
🆘 막히면: 파일이 빠졌나 → LLM에게 "만든 파일 목록 대조해줘".
📝 **핸드오프**: GitHub 저장소 URL, 올린 파일 목록.

### 페이즈 2 · Vercel 연결 + 키 등록 → 배포

🎯 **목표**: GitHub 저장소를 Vercel에 연결하고 키를 등록해 **공개 주소**를 받습니다.

🤖 **분기 A**: "이 저장소를 Vercel에 연결해 배포하고, 환경변수 4종(`NOTION_TOKEN`·`NOTION_DS_RUNNING`·`NOTION_DS_CHALLENGERS`·`GEMINI_API_KEY`)을 Vercel에 등록해줘. 끝나면 URL 알려줘."
💬 **분기 B**:
  1. vercel.com → (GitHub 계정으로 로그인) → **Add New → Project** → 페이즈 1 저장소 **Import**.
  2. 배포 전 **Environment Variables**에 키 4종을 이름=값으로 입력. **이 화면에만** 넣습니다(채팅 X).
  3. **Deploy** → 끝나면 `https://....vercel.app` 주소가 나옵니다.

✅ **완료 기준**: `https://....vercel.app` 공개 주소 확보.
📝 **핸드오프**: 배포 URL, Vercel에 등록한 키 **이름들**.

### 페이즈 3 · 라이브에서 검증 🎉

🎯 **목표**: 배포된 주소에서 실제로 도는지 확인합니다. **여기가 유일한 진짜 테스트.**

브라우저에서 순서대로:
1. `{내주소}/api/debug` 열기 → 키 4종이 **"등록됨"**으로 나오면 OK (값은 안 보임).
2. `{내주소}/api/challengers` 열기 → 챌린저 **이름들이 JSON으로** 나오면 노션 읽기 OK.
3. 메인 `{내주소}`를 **휴대폰**으로 열기 → 본인 고르고 러닝 스크린샷 한 장 업로드 → 노션 '러닝 기록'에 새 페이지가 채워지면 **완성.**

🆘 **막히면 (공통 디버그)**:
- `/api/debug`에 키 빠짐 → Vercel 환경변수 확인(페이즈 2). **키 추가/수정 후엔 재배포 필요**(Vercel → Deployments → ⋯ → Redeploy).
- `/api/challengers`가 비거나 오류 → 노션 '챌린저' DB에 사람이 있는지 + DB 2개에 인테그레이션을 **초대**했는지(페이즈 0).
- 업로드는 되는데 기록 안 됨 / 칸이 빔 → ① 노션 칸 이름이 §1과 **글자까지 같은지**, ② Vercel **함수 로그**(Deployments → 해당 배포 → Functions/Logs)의 오류 메시지를 **그대로 AI에게 보여주고** 고쳐달라고.

> **고치는 법**: 파일을 고쳐 GitHub에 반영하면(분기 A=push / 분기 B=GitHub 웹에서 파일 **Edit → Commit**) Vercel이 1분 내 **자동 재배포**합니다.

## 3. 핸드오프 프로토콜 (컨텍스트 한계 대비)

AI는 대화가 길어지면 앞부분을 잊거나 헤맵니다. 그래서 **페이즈마다 핸드오프를 남기고, 필요하면 새 세션에서 이어갑니다.**

### 페이즈를 마칠 때마다 — 핸드오프 갱신

분기 A는 에이전트가 저장소에 `HANDOFF.md`를 만들어 갱신합니다. 분기 B는 챗봇에게 **"지금까지의 핸드오프 요약을 만들어줘"**라고 해서 그 텍스트를 따로 보관(또는 GitHub에 `HANDOFF.md`로 저장)하면 됩니다. 담을 내용:

```markdown
# HANDOFF — 러닝 챌린지 빌드
- 분기: A / B
- 완료한 페이즈: 0,1 ... / 현재: 페이즈 2 진행 중
- GitHub 저장소: <URL>
- Vercel: 배포 URL=<...> · 등록한 키 이름: NOTION_TOKEN, ... (값은 적지 않음)
- 노션: 러닝 기록 DS=<id> · 챌린저 DS=<id>
- 다음 할 일: <다음 페이즈/작업>
- 막힌 점: <있으면>
```

> ⚠️ 토큰·키 **값**은 절대 적지 마세요. 이름만.

### 컨텍스트가 차면 — 새 세션에서 이어가기

1. 새 대화(세션)를 엽니다.
2. **분기 A**: 저장소 + `HANDOFF.md`를 에이전트에게 주고 → "`HANDOFF.md` 읽고 다음 페이즈부터 이어가줘."
   **분기 B**: ① 이 가이드 전체 → ② 핸드오프 내용 순서로 붙여넣고 → "여기부터 이어가자."
3. AI가 현재 상태를 요약하게 한 뒤 진행합니다.
