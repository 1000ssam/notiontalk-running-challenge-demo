# 02 · 워크플로 — 부품들이 어떻게 연결되나 (전체 지도)

[01 개념](./01-concepts.md)에서 단어를 익혔다면, 이제 그것들이 **어떻게 맞물려 하나로 움직이는지** 봅니다.
여전히 코드는 거의 없습니다. "무엇이 무엇을 호출하는가"의 흐름도입니다.

---

## 큰 그림 — 자동화는 사실 2개입니다

```
┌─ 자동화 1 · 입력 (이 가이드의 따라하기 범위) ───────────────────┐
│                                                                  │
│   [브라우저] public/index.html                                   │
│       │  ① 페이지 열면 챌린저 목록 요청                          │
│       │ ───────────────▶ /api/challengers ──▶ 노션 '챌린저' 표   │
│       │  ② 본인 선택 + 사진 업로드                               │
│       │ ───────────────▶ /api/upload                            │
│       │                      ├─ lib/gemini.js  사진 → 데이터(JSON)│
│       │                      └─ lib/notion.js  노션에 새 줄 + 사진 │
│       │ ◀─────────────── 결과 카드(추출값 + 노션 링크)           │
└──────────────────────────────────────────────────────────────────┘

┌─ 자동화 2 · 알림 (심화 · 개념만) ──────────────────────────────┐
│   노션에 줄 추가 ──웹훅──▶ /api/webhook/run-added → 즉시 문자     │
│   매일 아침   ──Vercel Cron──▶ /api/cron/daily-morning → 정기 문자│
│       └─ lib/solapi.js(발송) · messages.js(문구) · phone.js(번호) │
└──────────────────────────────────────────────────────────────────┘
```

> 따라하기([03](./03-tutorial.md))는 **자동화 1**까지 만듭니다. 자동화 2는 이 문서 아래쪽에서 구조만 설명합니다.

---

## 자동화 1의 생애주기 — 사진 한 장이 노션 데이터베이스 페이지가 되기까지

챌린저가 사진을 올린 순간, 안에서 이런 순서로 일이 벌어집니다:

1. **목록 채우기** — 페이지를 열면 브라우저가 `/api/challengers`에 "참가자 누구야?"라고 묻습니다.
   → 이 입구는 `lib/notion.js`의 `listChallengers()`로 노션 '챌린저' 표를 읽어 이름·아이콘을 돌려줍니다. 그래서 **노션에 챌린저를 한 줄 추가하면 웹 화면에 자동으로 나타납니다.**
2. **업로드** — 사용자가 본인을 고르고(한 번 고르면 브라우저가 기억) 사진을 첨부해 `/api/upload`로 보냅니다. (사진 + "누가 올렸는지" ID가 함께 전송됨)
3. **사진 읽기(AI)** — `/api/upload`는 사진을 `lib/gemini.js`에 넘깁니다. 여기서 Gemini가 화면을 읽고 **정해진 칸에 맞춘 데이터**를 돌려줍니다:

   ```
   { 거리, 시간(초), 페이스(초/km), 심박수, 고도, 케이던스, 칼로리, 날짜, 앱이름 }
   ```
   화면에 없는 값은 비어 있습니다(추측 안 함).
4. **노션에 기록** — `lib/notion.js`의 `createRunningRecord()`가 노션 '러닝 기록' 표에 **새 줄**을 만듭니다. 비어 있지 않은 값만 칸에 채우고, 사진은 인증샷으로 첨부하고, "누가" 칸은 1번에서 받은 ID로 연결합니다.
5. **응답** — 추출된 값과 노션 링크를 브라우저로 돌려주면, 화면에 "이렇게 기록됐어요" 카드가 뜹니다.

> 보이는 포인트: **AI가 닿는 곳은 딱 3번(사진 읽기) 한 군데**입니다. 나머지(목록 읽기·줄 만들기·사진 첨부·연결)는 전부 코드가 정해진 대로 처리합니다 — [01의 "LLM vs 스크립트"](./01-concepts.md#llm-vs-스크립트--이-가이드의-철학-축-) 그대로입니다.

## 파일별 역할 — 무슨 일을 하는 파일인가

코드를 읽을 필요는 없습니다. "이 파일은 이런 담당이구나" 정도만 알면 충분합니다.

### 자동화 1 (따라하기에서 만드는 것들)

| 파일 | 담당 |
|------|------|
| `public/index.html` | 사람이 보는 화면 — 챌린저 고르기 + 사진 올리기 + 결과 보기 |
| `api/challengers.js` | "참가자 목록 알려줘" 입구 |
| `api/upload.js` | "사진 받아서 처리해줘" 입구 (자동화 1의 심장) |
| `lib/gemini.js` | 사진 → 데이터(JSON). AI에게 줄 **지시문(프롬프트)** 과 **답 형식(칸 목록)** 이 여기 있음 |
| `lib/notion.js` | 노션과 대화하는 담당 — 목록 읽기, 새 줄 만들기, 사진 첨부 |
| `cli-upload.mjs` | (웹 만들기 전 점검용) 터미널에서 사진 한 장으로 진짜 되는지 테스트 |
| `api/debug.js` | "키들이 제대로 등록됐나?"만 확인 (값은 절대 안 보여줌) |

### 자동화 2 (심화 — 만들지 않고 구조만)

| 파일 | 담당 |
|------|------|
| `api/webhook/run-added.js` | 노션에 줄이 추가된 *순간* 호출 → 진행도/달성 축하 문자 |
| `api/cron/daily-morning.js` | 매일 아침 실행 → 독려/첫날/마지막날 문자 |
| `lib/solapi.js` | 실제 문자 발송 (문자 서비스 Solapi 호출) |
| `lib/messages.js` | 문자 문구 만들기 (노션 템플릿 표에서 문구를 가져옴) |
| `lib/phone.js` | 전화번호 형식 정리 (국내/해외 구분) |
| `lib/notion-challenge.js` | 문자용 노션 조회 모음 (러닝 횟수 세기, 전화번호 찾기 등) |

## 노션 데이터 모델 — 어떤 표를 미리 만들어 두나

자동화 1을 돌리려면 노션에 **표 2개**가 필요합니다. (표 = 노션 데이터베이스)

**① 러닝 기록 표** — 자동으로 채워지는 곳

| 칸 이름 | 유형 | 누가 채우나 |
|---------|------|-------------|
| 제목 | 제목(title) | 코드 (그날 날짜) |
| 일시 | 날짜(date) | 코드 |
| 챌린저 | 관계(relation) → 챌린저 표 | 코드 (누가 올렸는지 연결) |
| 인증샷 | 파일(files) | 코드 (올린 사진 첨부) |
| 거리(km), 시간(초), 평균 페이스(초/km), 평균 심박수, 고도 상승(m), 케이던스, 칼로리 | 숫자(number) | AI가 읽은 값 |
| 시간(표시), 평균 페이스(표시) | 수식(formula) | 노션이 자동 계산 (초 → "32:15" 보기 좋게) |

**② 챌린저 표** — 참가자 명단

| 칸 이름 | 유형 | 쓰임 |
|---------|------|------|
| 이름 | 제목(title) | 웹 화면 드롭다운에 표시 |
| (아이콘) | 페이지 아이콘 | 화면에 동그란 아바타로 표시 |

> **관계(relation)가 핵심**입니다. '러닝 기록'의 "챌린저" 칸이 '챌린저' 표와 연결돼 있어서, 누가 어떤 기록을 올렸는지 노션이 자동으로 묶어줍니다. (양방향으로 걸어두면 챌린저 쪽에서도 "이 사람의 기록들"이 보입니다.)

## 심화 — 문자 자동화의 안전장치들

> 자동화 2를 나중에 만들 때 알아두면 좋은 개념. 지금은 "이런 게 있구나" 정도로 충분합니다.

문자는 잘못 보내면 곤란하므로(같은 사람에게 두 번, 챌린지 끝났는데 또…) 안전장치가 들어갑니다:

- **멱등성(중복 방지)**: 같은 기록에 대해 문자가 이미 나갔으면 다시 안 보냅니다. (러닝 기록에 "문자발송완료" 표시를 남겨두고 확인)
- **발송 on/off 스위치(노션으로 제어)**: 노션 '챌린지 정보' 페이지의 토글 하나로 전체 문자 발송을 켜고 끕니다. 코드를 다시 배포할 필요 없이, **노션에서 스위치만 내리면 멈춥니다.** (정보가 불확실하면 *안 보내는* 쪽으로 안전하게 동작)
- **재시도**: 노션이 "너무 자주 부른다(429)"고 하면 잠깐 기다렸다 다시 시도합니다.
- **국내/해외 구분**: 해외 번호는 짧은 문자(SMS)로, 이모지·제목 없이 보냅니다.

이 모든 게 [01의 "LLM vs 스크립트"](./01-concepts.md#llm-vs-스크립트--이-가이드의-철학-축-)에서 말한 *"정해진 판단은 코드가 결정적으로"* 의 실제 예시입니다.
