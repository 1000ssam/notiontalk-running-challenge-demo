# 01 · 개념 — 낯선 단어를 비유로 이해하기

이 자동화를 만들 때 계속 나오는 단어들을 **비유 중심**으로 풀었습니다.
코드는 한 줄도 없습니다. 각 개념마다 *"이 프로젝트에서는 어디에 쓰였나"* 를 붙여뒀으니, 따라 하다 막히는 단어가 나오면 여기로 돌아오세요.

> 한 문장 요약: **이 프로젝트는 "AI가 잘하는 일(사진 읽기)"만 AI에게 맡기고, 나머지(노션에 넣기·문자 보내기 같은 정해진 일)는 코드가 자동으로 처리합니다.**

---

## API — 프로그램끼리 주고받는 "전용 창구"

- **한 줄 정의**: 사람이 아니라 *프로그램*이 다른 서비스에 부탁할 때 쓰는 정해진 창구.
- **비유**: 식당에 입구가 두 개 있다고 생각해보세요.
  - 사람용 입구 = 메뉴판 보고 손으로 주문 (= 우리가 노션 사이트에서 직접 타이핑하는 것)
  - 배달앱 전산 연동 입구 = 프로그램이 자동으로 주문 넣음 (= **API**)
- **이 프로젝트에서는**: 우리 코드가 노션 사이트(notion.so)가 아니라 **노션 API(api.notion.com)** 라는 창구로 들어가서, 자동으로 표에 새 항목을 만듭니다. (노션에선 표의 이 한 줄 한 줄을 '**데이터베이스 페이지**'라고 부릅니다 — 이 가이드도 그렇게 부릅니다.) Gemini(이미지 AI)도, 문자 발송(Solapi)도 전부 API로 부탁합니다.

> 핵심 한 줄: 어떤 서비스든, 어떤 언어로 짜든, 결국은 **"API에 요청 보내고 → 답(데이터)을 받는 것"** 의 반복입니다.

---

## 인테그레이션 & 토큰 — "출입 사원증과 비밀번호"

- **한 줄 정의**: 인테그레이션은 노션이 발급하는 *출입 자격*, 토큰은 그 자격을 증명하는 *비밀번호 문자열*.
- **비유**:
  - 인테그레이션 = 회사가 발급한 **출입 사원증**
  - 토큰(`ntn_...` 같은 문자열) = 사원증의 **비밀번호**
  - 사원증이 있어도 아무 방이나 못 들어갑니다 → **들어갈 방(=노션 DB)을 따로 지정**해야 합니다. (노션 DB 우상단 `⋯ → 연결 추가`)
- **이 프로젝트에서는**: "클로드 코드"라는 인테그레이션을 만들고, 그 토큰으로 우리 코드가 노션에 들어갑니다. 토큰이 없으면 노션은 "누구세요?" 하며 거절합니다(401 오류).

---

## 시크릿 관리 — "비밀번호는 따로 보관"

- **한 줄 정의**: 토큰·API 키처럼 노출되면 안 되는 값(=시크릿)은 코드와 *분리해서* 보관한다.
- **비유**: 일기장(코드)에 통장 비밀번호를 적지 않습니다. 비밀번호는 따로 적어 금고에 둡니다.
- **이 프로젝트에서는** 시크릿을 두 곳에만 둡니다:
  - 내 컴퓨터: **`.env.local`** 파일 (← `.gitignore`에 등록해서 깃허브에 안 올라가게 막음)
  - 배포 서버(Vercel): Vercel 설정에 따로 등록
- **왜 안전한가**: 깃허브에 코드를 통째로 공개해도 **시크릿은 거기 없습니다.** 코드와 비밀번호가 다른 채널로 흐르기 때문입니다.

> ⚠️ 가장 중요한 규칙: **토큰·키를 채팅창에 붙여넣거나 깃허브에 올리지 마세요.** 한 번 노출되면 누구나 여러분 이름으로 노션·문자를 조작할 수 있습니다.

---

## 비전 OCR & 구조화 출력 — "사진을 읽고, 정해진 칸에 맞춰 답하기"

- **한 줄 정의**: 이미지를 이해하는 AI(비전 LLM)에게 스크린샷을 보여주고, 원하는 항목만 **정해진 형식(JSON)** 으로 받아내는 것.
- **옛날 방식 vs 지금**:
  - 옛날(전통 OCR): 글자 위치를 좌표로 일일이 잡고 정규식 수백 줄 → 앱마다 새로 짜야 함
  - 지금(비전 LLM): "이 화면에서 거리·시간·페이스를 뽑아줘"라고 **말로 부탁** → 끝
- **구조화 출력이 왜 중요한가**: AI에게 그냥 물으면 답을 문장으로 줍니다("약 5km 정도 뛰셨네요!"). 그러면 프로그램이 못 씁니다. 그래서 **"반드시 이런 칸(거리, 시간, 페이스…)으로만 채워서, JSON 형식으로 답해"** 라고 형식을 강제합니다. → 답을 곧바로 노션에 넣을 수 있습니다.
- **프롬프트의 안전장치**: "화면에 안 보이는 값은 **추측하지 말고 비워둬(null)**" 라고 지시합니다. 안 그러면 AI가 그럴듯한 가짜 숫자를 채워 넣습니다.
- **이 프로젝트에서는**: 애플워치·가민·삼성헬스·NRC 등 제각각 생긴 화면을 같은 AI 한 번으로 처리합니다. 단위(마일→km, "32:15"→1935초)도 AI가 변환합니다.

## LLM vs 스크립트 — 이 가이드의 철학 축 🔑

여기가 가장 중요한 개념입니다. **"왜 전부 AI에게 안 시키고, 코드를 같이 쓰나?"**

- **LLM(브라우저 챗봇)** 만으로는 두 가지 한계가 있습니다:
  1. **매번 처음부터 해석합니다.** "거리 5km 넘으면 A, 아니면 B" 같은 단순 판단도 매번 AI에 물으면 느리고, 비싸고, 답이 매번 살짝 다를 수 있습니다.
  2. **진짜 일을 못 합니다.** 챗봇은 *텍스트*만 만듭니다. 파일을 만들거나 노션에 데이터베이스 페이지를 추가하거나 문자를 보내지 못합니다.
- **스크립트(코드)** 는 이 둘을 보완합니다:
  1. 분기·반복·조건을 **100% 똑같이, 즉시, 거의 무료로** 처리합니다.
  2. 파일 읽기·API 호출·DB 입력 같은 **컴퓨터의 진짜 능력**을 씁니다.

- **비유로 정리**:

| | 정체 | 한 줄 비유 |
|---|---|---|
| 브라우저 챗봇 LLM | 매번 시키는 조수 | 매번 처음부터 듣고 텍스트만 답함 |
| 스킬(skill.md) | 매뉴얼 읽는 조수 | 일관성↑, 그래도 매번 조수 시간·비용 듦 |
| 스크립트(.mjs/.py) | 매뉴얼대로 도는 **기계** | 한 번 만들면 조수 없이 영구히 돎 |
| **클로드 코드** | 그 기계를 **말로 부탁하면 짜주고 돌려주는 도구** | 자연어 요청 → 코드 작성 → 실제 실행 |

- **이 프로젝트의 구조가 정확히 그것**:
  - **추론(AI에게)**: 사진 → 러닝 데이터 뽑기 (이것만 Gemini)
  - **정해진 흐름(코드가)**: 누가 올렸는지 매칭, 파일 형식 분기, 값 정리
  - **실제 동작(코드가)**: 사진 파일 읽기, 노션 API 호출, 표에 데이터베이스 페이지 추가

> 한 줄 메시지: **AI는 "판단·읽기"에만, 코드는 "정해진 일·실제 동작"에. 합치면 빠르고·안정적이고·거의 무료입니다.**

---

## 서버리스 함수 — "필요할 때만 깨어나는 작은 일꾼"

- **한 줄 정의**: 평소엔 잠들어 있다가, 요청이 오면 깨어나서 일 한 번 하고 다시 잠드는 작은 프로그램.
- **비유**: 24시간 켜둔 서버(상주 직원)가 아니라, 호출벨을 누를 때만 오는 **콜택시**.
- **이 프로젝트에서는**: `api/` 폴더 안의 **파일 하나가 곧 입구(주소) 하나**입니다.
  - `api/upload.js` → "사진 받아서 처리하는 입구"
  - `api/challengers.js` → "챌린저 목록 알려주는 입구"
  - 누가 그 주소로 요청을 보낼 때만 잠깐 실행됩니다 → 평소엔 비용 0.

## 배포 & Vercel — "내 노트북이 꺼져도 돌아가게"

- **배포(deployment)란**: 내 노트북에서만 돌던 코드를 *인터넷의 어떤 컴퓨터*에 올려서 누구나 접속하게 만드는 것. = **"내 노트북을 꺼도 돌아가는 상태"**.
- **Vercel이란**: "코드만 올리면 알아서 굴려주는" 클라우드 서비스. 깃허브에 코드를 올리면 **자동으로 빌드해서 인터넷 주소(URL)를 줍니다.** 개인 학습용은 무료.
- **출판사 비유**:
  - 깃허브 = 원고 보관 **창고**
  - Vercel = 받아서 인쇄·배포하는 **출판사**
  - 푸시(업로드) = 새 원고 **입고**
  - 배포 = 책이 서점에 **진열**
  - URL = ISBN, 누구나 그걸로 찾아옴
- **이 프로젝트에서는**: 깃허브에 코드를 올리면 Vercel이 받아 `https://...vercel.app` 주소를 만들어줍니다. 그 주소를 챌린저들에게 알려주면 끝.

> 보너스: 깃허브에 코드를 새로 올릴 때마다 Vercel이 **1분 안에 자동 재배포**합니다. "코드 고쳐서 올림 = 라이브 즉시 반영".

## (심화) 웹훅 vs 크론 — 자동 문자를 보내는 두 가지 방아쇠

> 이 두 개념은 **심화 기능(자동 문자 발송)** 에 쓰입니다. 입력 자동화만 만들 거라면 지금은 건너뛰어도 됩니다. ([03 따라해보기](./03-tutorial.md)는 입력 자동화까지만 다룹니다.)

- **웹훅(Webhook) = "사건이 생기면 즉시 알림"**
  - 비유: 택배가 도착하면 누르는 **초인종**. 노션에 새 러닝 기록이 생기는 *순간* 우리 코드를 호출합니다.
  - 이 프로젝트: 기록이 추가되면 → 해당 챌린저에게 "오늘도 달리셨네요!" 진행도 문자 즉시 발송.
- **크론(Cron) = "정해진 시각마다 알아서"**
  - 비유: 매일 같은 시각에 울리는 **알람 시계**. 사건과 무관하게 *시간이 되면* 실행됩니다.
  - 이 프로젝트: 매일 아침, 며칠 쉰 챌린저에게 독려 문자 / 챌린지 첫날·마지막날 안내 문자 발송.

이 두 방아쇠를 받쳐주는 추가 개념(멱등성=중복 발송 방지, 발송 on/off 스위치 등)은 [02 워크플로](./02-workflow.md#심화--문자-자동화의-안전장치들)에서 짧게 정리했습니다.
