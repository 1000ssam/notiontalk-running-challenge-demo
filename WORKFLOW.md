## 🎯 큰 서사 (4단계 흐름)

청중에게 보여줄 흐름은 다음 4단계로 흘러간다:

1. **OCR 로직 짜기** — 이미지 → JSON 두뇌
2. **노션 입력 스크립트 짜기** — 결과를 노션에 던지는 손
3. **클코에서 시연** — `node cli-upload.mjs` 한 줄로 진짜 돌아가는지 확인
4. **버셀에 배포** — CLI 못 치는 사용자를 위해 웹앱으로 감싸기

각 단계 멘트 한 줄:
- 1: "사진을 JSON으로 바꾸는 두뇌를 만듭니다."
- 2: "그 결과를 노션에 던지는 손을 만듭니다."
- 3: "그래서 진짜로 되는지 한 번 돌려봅니다."
- 4: "근데 사용자는 CLI 못 치니까 웹으로 감쌉니다."

### 🔑 핵심 메시지: "스크립트"가 LLM의 한계를 넘는 이유

1→2로 넘어가는 사이에 **반드시 박을 한 컷**. 이게 오늘 시연의 가장 중요한 메시지.

#### 먼저 — 스킬과 스크립트의 차이부터

청중이 헷갈릴 수 있는 두 개념을 먼저 갈라준다.

| | **스킬 (skill.md)** | **스크립트 (.mjs/.py)** |
|---|---|---|
| 형식 | 자연어 마크다운 | 프로그래밍 언어 |
| 실행하는 주체 | **LLM(클로드)이 읽고 따라함** | **컴퓨터가 직접 실행** |
| 실행마다 | LLM 호출 필요 (토큰 소모) | LLM 없이 돌아감 |
| 결정성 | LLM 변동 영향 받음 | 100% 동일 결과 |
| 속도 | 수~수십 초 | 밀리초 |
| 비용 | API 토큰 | 거의 0 |
| 적합한 작업 | **판단·추론 필요** | **정해진 절차** |

> 핵심: 스킬도 "워크플로 정의"는 가능. 차이는 **실행 주체**. 스킬은 매번 LLM이 읽고 해석. 스크립트는 LLM 없이 컴퓨터가 그대로 실행.

#### 그래서 스크립트의 본질 — 두 축으로 LLM을 보완한다

**축 1. LLM 추론에 의존하지 않는 흐름을 삽입한다**
- 분기·반복·조건은 코드가 **결정적**으로 처리
- 예: "거리 5km 넘으면 A, 아니면 B" → 매번 LLM에 묻지 않음. `if (distance > 5)` 한 줄.
- LLM은 통계적·확률적 → 같은 입력도 다를 수 있음. **코드는 100% 동일**.
- 효과: 안정성·재현성·속도·비용 압도적 우위

**축 2. 실제로 프로그램을 동작시킬 수 있다**
- 파일 읽고 쓰기, API 호출, DB 입력, 네트워크 통신, 시스템 명령 — **컴퓨터의 진짜 능력**
- LLM 단독(브라우저 챗봇)은 텍스트만 생산. 진짜 동작은 못 함.
- 클로드코드는 **LLM이 코드를 짜고 → 그 코드가 컴퓨터에서 진짜 일을 함**
- 효과: 해당 컴퓨터의 모든 능력 활용 (파일·DB·API·하드웨어까지)

**두 축을 합치면**: LLM은 추론이 필요한 부분(이미지 분석)에만 호출되고, 나머지 결정적 흐름과 실제 동작은 스크립트가 담당. **안정적이면서 진짜 일을 함**.

**우리 파이프라인의 구조가 정확히 그것**:
- **추론 필요 (LLM)**: 이미지 → 러닝 데이터 JSON 추출 (Gemini)
- **결정적 흐름 (스크립트, 축 1)**: 챌린저 ID 매칭, 파일 형식 분기, JSON 파싱
- **실제 동작 (스크립트, 축 2)**: 이미지 파일 읽기, base64 변환, HTTP API 호출, 노션 행 생성

#### 시연 멘트

> "LLM은 추론은 잘해요. 근데 두 가지 한계가 있습니다.
>
> **첫째, 매번 LLM이 모든 단계를 해석해야 합니다**. '거리 5km 넘으면 A 챌린저, 아니면 B 챌린저' 같은 결정도 매번 LLM에 묻는다? 느리고 비싸고 매번 다를 수도 있어요. **이런 결정은 코드 한 줄이면 끝**(`if distance > 5`)이고, 매번 똑같이 작동합니다.
>
> **둘째, 브라우저 챗봇 LLM은 진짜 일을 못 해요**. 파일을 만들거나 노션 API를 호출하거나 폴더를 정리하지 못합니다. 텍스트만 만들죠. 클로드코드가 다른 점은 **LLM이 코드를 짜주고 → 그 코드가 진짜로 컴퓨터에서 돌아간다**는 거예요.
>
> 그래서 우리 파이프라인은 LLM이 잘하는 한 부분(이미지 분석)에만 LLM을 쓰고, 나머지(분기·파일·API·노션 입력)는 모두 스크립트가 결정적으로 처리합니다. **빠르고, 안정적이고, 거의 무료**. 이게 LLM 한계를 넘는 진짜 이유."

#### 비유

- **LLM 단독** (브라우저 챗봇) = 조수에게 매번 일일이 시키는 것. 매번 처음부터 듣고 해석.
- **스킬** = 조수가 매번 매뉴얼을 읽고 따라하는 것. 매뉴얼 있으니 일관성↑, 그래도 매번 조수의 시간·비용은 든다.
- **스크립트** = 그 매뉴얼대로 작동하는 **기계**. 한 번 만들면 조수 없이도 알아서 돌아감.
- **클코** = 그 기계를 사용자가 자연어로 부탁하면 짜주고 실행까지 해주는 도구.

#### LLM(브라우저 챗봇) 한계 정리 (시연 시 한 장)

| | 브라우저 챗봇 LLM | 클코 + 스크립트 |
|---|---|---|
| 컴퓨터 조작 | 텍스트만 보여줌 | 파일 만들고·실행 |
| 같은 일 반복 | 매번 처음부터 | **한 번 짜면 영구** |
| 비용/속도 | 매번 토큰·초 단위 | **거의 무료·밀리초** |
| 결과 일관성 | 매번 다를 수 있음 | **결정적** |
| 시크릿 보관 | 위험 | 환경변수로 안전 |
| 외부 서비스 연동 | 직접 못 함 | API·웹훅 자유 |

## 1. 노션 API 연결
- 목적: 우리 스크립트가 노션 워크스페이스를 조작할 수 있게 권한 통로 확보
- 사전 준비물:
  - "클로드 코드" 인테그레이션이 노션에 미리 등록되어 있음
  - 토큰: `.env.local`의 `NOTION_TOKEN`으로 주입 (`ntn_...` 형식, 절대 커밋 금지)
  - 노션 호출 헬퍼: `lib/notion.js` (또는 공식 `@notionhq/client`)
- 명령어 / 입력:
  - 인테그레이션을 시연 DB에 초대 (노션 DB 우상단 ⋯ → "연결 추가" → "클로드 코드" 선택)
- 예상 결과: 스크립트에서 토큰 자동 로드되어 호출 시 200 OK
- 막히면:
  - 401/403: DB에 인테그레이션 추가 안 됨 → 다시 초대
  - "Not found": DB ID 오타 (URL의 32자리 16진수 부분)

### 청중 시연 흐름

#### (1) 노션 API 공식 문서 보여주기
- URL: <https://developers.notion.com/reference/intro>
- 멘트: "노션은 사람용 사이트(notion.so)와 **별개로** 프로그램이 들어오는 입구가 있습니다. 그게 **api.notion.com**, 그리고 이 페이지는 그 입구의 **사용설명서**예요."
- 강조 포인트: Base URL · HTTP 메서드(GET/POST/PATCH) · 인증 헤더 두 줄(`Authorization`, `Notion-Version`)

#### (2) API가 뭔지 설명
- "**A**pplication **P**rogramming **I**nterface — 프로그램이 다른 프로그램한테 부탁할 때 쓰는 정해진 창구."
- 비유: 식당의 두 입구 (사람용 메뉴판 vs 배달앱 전산 연동)
- 노션 적용: "내가 손으로 글 쓰는 건 사람용 입구. 우리 스크립트가 자동으로 글 쓰려면 API 입구."

#### (3) curl이 뭔지 설명
- "터미널에서 URL 한 번 호출해주는 **만능 망치**. API 동작 확인할 때 1초 만에 씀."
- 핵심 옵션 4개만 외워두면 됨:
  - `-X GET` 읽기 / `-X POST` 쓰기 / `-X PATCH` 수정 / `-X DELETE` 삭제
  - `-H` 헤더 추가 (사원증 보여주기)
- 헤더 두 줄의 의미:
  - `Authorization: Bearer ntn_...` → "이 사원증으로 들어왔습니다"
  - `Notion-Version: 2026-03-11` → "이 시점 규격으로 대화합니다"

#### (4) 실제 curl 1줄 시연 (라이브)
```bash
curl -X GET 'https://api.notion.com/v1/databases/<DB_ID>' \
  -H 'Authorization: Bearer ntn_...' \
  -H 'Notion-Version: 2026-03-11'
```
- 결과: 노션이 JSON으로 DB 메타데이터(`title: 러닝 기록` 등) 뱉음
- 멘트: "보세요, 명령어 한 줄 = 노션 데이터 한 덩어리. 우리가 짠 노드 스크립트도 결국 이걸 자바스크립트로 포장한 것."

#### (4-1) 언어 전환 보여주기 (docs 코드 블록 우상단 드롭다운)
- docs에 자바스크립트/컬/파이썬/PHP/Java/Ruby/Go 다 들어있음
- 멘트: "**같은 노션 API를 부르는 7가지 언어 버전이에요. 본질은 다 똑같음**. 어떤 언어를 쓰든 결국 같은 노션이랑 대화합니다."
- curl vs 자바스크립트 비유:
  - **curl** = 동네 슈퍼 우유 한 팩 사 오는 일회성 심부름
  - **자바스크립트** = 일주일치 장보기 + 요리 + 정리까지 하는 가정부
- 우리 프로젝트는 **자바스크립트** 선택 (Node.js + Vercel 궁합)

#### (5) 인테그레이션·토큰 비유
- 인테그레이션 = 노션이 발급하는 "출입 사원증"
- 토큰 = 사원증의 비밀번호 (`ntn_...`)
- 사원증이 있어도 출입 가능 구역(=DB)은 따로 지정 → "DB ⋯ → 연결 추가"
- 토큰은 절대 깃허브/대화창에 노출 금지 → `.env` + `.gitignore`

## 3. 이미지 OCR (Gemini Vision)
- 목적: 러닝 앱 스크린샷에서 거리·시간·페이스·심박 등을 JSON으로 자동 추출
- 사전 준비물:
  - Gemini API 키 (`.env.local`의 `GEMINI_API_KEY`)
  - 모델: `gemini-2.5-flash` (`gemini-2.0-flash`은 deprecated)
  - 테스트 이미지 (`samples/`)
- 명령어 / 입력:
  - 스크립트: `node test-ocr.mjs`
  - 핵심: HTTP POST `https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash:generateContent?key=KEY`
  - 페이로드: `contents.parts`에 프롬프트 + `inline_data`(base64 이미지)
  - **JSON 강제**: `generationConfig.response_mime_type = "application/json"` + `response_schema`
- 예상 결과: 스키마에 100% 일치하는 JSON. 6장 모두 파싱 실패 0건.
- 막히면:
  - `model not available`: 새 모델명으로(`gemini-2.5-flash`)
  - 값이 0으로 채워짐: required에서 빼고 nullable로
  - 화면에 안 보이는 값을 추측해서 채움: 프롬프트에 "추측 금지·null로" 명시
- 청중 시연 포인트:
  - "전통 OCR vs 비전 LLM" 비교 — 옛날: 정규식 수백 줄 / LLM: 프롬프트 + 스키마 = 끝
  - JSON 스키마가 강제되어 결과 파싱 실패 0%
  - "코드 = 매니저, LLM = 외국 번역가" 비유 (둘은 다른 컴퓨터, HTTP로 대화)

## 4. 자동화 시나리오 설계 → 웹 업로드 페이지 + 서버 함수

### 🎯 시연자 메모: 이 단계는 **플랜모드**로 진입할 것
- `Shift+Tab` 두 번 → "Plan mode" 활성화
- 청중에게 "클로드는 코드부터 안 짜요. 먼저 **계획을 세웁니다**" 한 컷
- 플랜모드에서 자동화 시나리오를 짜고 → 사용자(시연자)가 검토 → 승인 후 빌드
- 메시지: "**진짜 일은 코딩이 아니라 결정**. 클로드가 이 결정을 같이 해줍니다."

### 결정 흐름 (시연 콘텐츠)

#### 입력 방식 결정 — 노션 직접 vs 웹 한 장
| | 노션에 직접 인증샷 첨부 | 웹페이지 한 장에서 업로드 |
|---|---|---|
| 켜는 무게 | 노션 워크스페이스 전체 | 한 화면 |
| 학습 곡선 | 노션 사용법 학습 필요 | 가르칠 게 0 |
| 시연 임팩트 | 잔잔 | 와우 (사진 → 노션 행 즉시 채워짐) |
| **선택** | | ✅ |

→ 메시지: "노션은 **결과 저장소**, 사용자 입구는 **가벼운 한 장**."

#### 사용자 식별 방식 결정 — 드롭다운 vs 개인 URL
| | 드롭다운 | 개인 URL |
|---|---|---|
| 운영자 안내 | "이 URL 가서 본인 골라요" 한 줄 | 챌린저별 URL 발급 |
| 챌린저 추가 | 노션에 한 줄 → 자동 반영 | URL 새로 발급+발송 |
| 빌드 복잡도 | 단일 페이지 | dynamic route 필요 |
| **선택** | ✅ (LocalStorage 캐싱으로 매번 선택 마찰 0) | |

→ 메시지: "**이런 트레이드오프 판단이 진짜 일**. 코딩보다 결정이 중요."

## 5. 깃허브 푸시
- 목적: 코드 백업 + 협업 가능 + Vercel이 받아갈 원천 마련
- 사전 준비물: gh CLI 로그인 (`gh auth status`), `git init` + 첫 커밋 완료
- 명령어:
  - **자연어로 시키기**: "이 코드 깃허브에 비공개로 올려줘" / "리포 만들고 푸시해줘"
  - 클로드가 실행: `gh repo create <이름> --private --source=. --remote=origin --push`
- 예상 결과: https://github.com/1000ssam/running-challenge-3 같은 URL 출력 + main 푸시 완료
- 막히면:
  - `gh: command not found` → gh CLI 설치
  - 401 → `gh auth login`
  - 같은 이름 리포 존재 → 다른 이름

### 청중 시연 핵심 — 6개 개념 정리

| 개념 | 한 줄 정의 | 비유 |
|---|---|---|
| **리포지토리** | 프로젝트 폴더 + 모든 변경 이력 | 책 + 수정 이력 |
| **커밋** | 변경 한 묶음 + 메시지 | 게임 세이브 포인트 |
| **푸시** | 로컬 → 깃허브 업로드 | 노트북 작업물 클라우드 백업 |
| **브랜치** | 동시 여러 작업 라인 | 평행우주 |
| **머지** | 두 브랜치 합치기 | 평행우주 통합 |
| **PR** | "내 브랜치 합쳐줘" 요청 + 토론 | 출판사 원고 검토 |

### .git 폴더 (자주 나오는 질문)
- 프로젝트 폴더 안 `.git/` 숨김 폴더에 모든 이력 저장 (1MB 안팎)
- HEAD/config/refs는 텍스트, objects/는 압축 바이너리
- `.git/` 지우면 이력만 사라지고 코드는 남음
- 깃허브 푸시 = `.git` 내용을 인터넷 서버에 복사

### .gitignore 헷갈림 정리
- `.gitignore`에는 **시크릿 값이 아니라 시크릿 파일 이름**이 들어감
- "출입금지 명단"일 뿐, 실제 시크릿은 `.env.local` 같은 별도 파일에
- 검증: `git check-ignore -v .env.local`

### PR이 솔로엔 사치
- 협업할 때 진가 (리뷰·토론·승인 한 곳에 모임)
- 혼자면 **커밋 메시지 잘 쓰기**가 더 큰 효과
- "좋은 커밋 메시지 = 미래의 자기에게 보내는 편지"

### 자연어 vs 명령어 (시연 한 컷)
> "**'init', 'commit', 'push'를 외울 필요 없어요. '이 코드 깃허브에 올려줘' 한 마디면 됨.** 클로드가 시크릿 점검 → `git init` → 첫 커밋 → 깃허브 리포 생성 → 푸시까지 분해해서 자동 실행."

## 6. Vercel 배포 + env

### 청중 시연 핵심 — "배포가 뭐고 Vercel이 뭔지" 먼저

#### 배포(deployment)가 뭔가요?
- 내 노트북에서만 돌던 코드를 **인터넷 어딘가의 컴퓨터에 올려서** 누구나 접근 가능하게 만드는 것
- **"내 노트북 꺼져 있어도 돌아가는 상태"**가 되는 것
- 비유: 내 책상 위 메모 → 회사 게시판에 붙임 (누구나 와서 볼 수 있음)

#### Vercel이 뭐 하는 회사인가요?
- **"코드만 올리면 알아서 굴려주는 클라우드 서비스"**
- 깃허브 리포 연결 → main 푸시할 때마다 **자동 빌드 + 자동 배포**
- 정적 파일(html) + 서버리스 함수(api/*) 둘 다 호스팅
- HTTPS·CDN·도메인 자동, 개인 학습용 **무료**

#### 옛날 vs 지금
| | 옛날 (자체 서버) | 지금 (Vercel 같은 클라우드) |
|---|---|---|
| 무엇을 사는가 | **컴퓨터 + 전기 + 냉방 + 보안 + 24시간 관리자** | 코드만 올림, 나머지 관리는 회사가 |
| 시작 비용 | 수백~수천만 원 | 0원 (무료 플랜) |
| 시작 시간 | 수개월 | 5분 |
| 코드 한 줄 고치면 반영까지 | 직접 서버 들어가서 수동 | **푸시 한 번 = 1분 이내 자동** |

#### "출판사" 비유 (시연용 한 컷)
- 깃허브 = 원고 보관 **창고**
- Vercel = 받아서 인쇄·배포해주는 **출판사**
- 푸시 = 원고 새 버전 **입고**
- 배포 = 책이 서점 **진열**
- URL = ISBN — 누구나 그걸로 찾아옴

#### 왜 Vercel을 골랐나? (대안 비교)
| 서비스 | 특징 |
|---|---|
| **Vercel** ⭐ | 정적+함수, 깃허브 연동, 가장 단순 |
| Netlify | Vercel과 거의 같음 (선호 차이) |
| AWS/GCP | 본격 클라우드, 학습 곡선 높음 |
| 직접 서버(VPS) | 자유도↑, 관리 책임↑ |

→ 입문자·소규모엔 **Vercel 또는 Netlify가 정답**.

#### 시연 멘트 흐름 (4컷)
1. "코드를 짰어요. 근데 **내 노트북에서만 돌아가요. 다른 사람은 못 씀**."
2. "그래서 인터넷의 어떤 컴퓨터에 우리 코드를 올려야 하는데, 그게 **배포**예요."
3. "Vercel이 그걸 가장 쉽게 해주는 서비스. **깃허브 리포 연결 → 푸시 → 자동 배포**. 5분이면 끝."
4. "받은 URL은 누구나 접속 가능. **내 노트북 꺼져 있어도 돌아갑니다**. 진짜 자동화 완성."

---

- 목적: 인터넷에서 접근 가능한 진짜 URL 받기 → 폰에서도 사용 가능
- 사전 준비물:
  - vercel CLI 로그인 (`vercel whoami`)
  - 깃허브에 푸시 완료 (Vercel이 거기서 받아감)
  - `.env.local`에 4개 시크릿 (NOTION_TOKEN, GEMINI_API_KEY, NOTION_DS_RUNNING, NOTION_DS_CHALLENGERS)
- 명령어 (3단계):
  ```bash
  # 1. Vercel 프로젝트 연결 (자동으로 깃허브 리포도 연결)
  vercel link --yes

  # 2. env 4개를 production에 등록 (시크릿 보안 패턴: printf 파이프 사용)
  for key in NOTION_TOKEN GEMINI_API_KEY NOTION_DS_RUNNING NOTION_DS_CHALLENGERS; do
    value=$(grep "^${key}=" .env.local | cut -d= -f2-)
    printf "%s" "$value" | vercel env add "$key" production
  done

  # 3. 프로덕션 배포
  vercel --prod
  ```
- 예상 결과:
  - Production URL (긴 버전): `https://<프로젝트>-<해시>-<팀>.vercel.app`
  - Aliased URL (짧은 버전): `https://<프로젝트>.vercel.app`  ← 이걸 청중에 알림
- 막히면:
  - env 값에 줄바꿈 들어가면 토큰 깨짐 → `printf "%s"` 사용 (`echo` 금지)
  - 401 Unauthorized: env가 production에만 있고 preview엔 없을 때 → preview까지 등록
  - "No framework detected": 정상. public/ + api/ 자동 인식

### 청중 시연 핵심 — 시크릿이 깃허브에 안 올라간 이유 한 컷

> "보세요, **시크릿은 두 곳에 따로 보관**됩니다.
> - 내 노트북 → `.env.local` (`.gitignore`에 등록되어 깃허브에 안 올라감)
> - Vercel 서버 → `vercel env add`로 별도 등록
>
> **그래서 깃허브 리포는 텅 비어 있어도 Vercel은 시크릿 알고 있음**. 둘은 다른 채널.
> 다른 사람이 우리 깃허브 다 봐도 시크릿 1글자도 안 보임."

### 배포 후 자동화의 미래 (안 박아도 되지만 멘트로 한 컷)
- 깃허브 자동 연동됨 → main에 푸시할 때마다 Vercel이 자동 재배포
- 즉 **"코드 고쳐서 푸시 = 라이브 즉시 반영"**
- 청중에 "이게 진짜 무서운 부분" 한 마디 (=잘못된 코드 푸시하면 라이브 깨짐 → 그래서 협업 시 PR로 검증)

### 검증
- 받은 URL을 폰 브라우저로 열기
- 챌린저 드롭다운이 노션에서 fetch한 5명 (1000쌤·닝닝·윈터·지젤·카리나)으로 채워지는지 확인
- 사진 한 장 업로드 → 30초 내 노션에 행 추가되는지 확인

## 7. (선택) 라이브 교재 — MD→HTML 자동 렌더 + 듀얼 화면

### 🎯 시연 콘셉트
시연 중에 우리가 무언가를 가르치거나 정리할 때, **클로드가 마크다운을 작성하면 옆 브라우저에서 그 내용이 1초 안에 자라나는 교재**로 변신하는 구조. "왼쪽 터미널 = 클로드 작업 / 오른쪽 브라우저 = 자라나는 교재" 듀얼 화면 임팩트.

> 핵심 메시지: **"같은 시간선에서 두 개가 동시에 움직입니다. 출판사 인쇄가 1초 안에 끝나는 거예요."**

### 왜 풀 HTML 매번 재작성이 아닌가?
풀 HTML로 매번 재작성하면 토큰 4~5배 비싸지고 Edit이 nested 구조 진입할 때마다 정밀도 떨어짐. **MD로 작업 + HTML 자동 렌더 = 같은 시각 임팩트, 토큰 비용 거의 0**.

### 파일 구조 (이미 생성됨, 위치: `/mnt/c/dev/notes/live-tutorial/`)
```
/mnt/c/dev/notes/live-tutorial/
  ├─ render.mjs       # fs.watchFile로 MD 감지 → marked로 HTML 변환
  ├─ tutorial.md      # 마크다운 원본 (여길 편집)
  ├─ tutorial.html    # 자동 생성 (브라우저로 직접 open)
  └─ package.json     # marked 단일 의존성
```

> 라이브 교재 도구는 **러닝 챌린지 프로젝트 자산이 아니라 범용 시연 도구**. 그래서 시연 폴더가 아니라 `/mnt/c/dev/notes/` 아래 별도 위치. 시연 폴더는 빈 채로 시작해야 "M0. 빈 폴더에서" 컨셉이 살아남.

### 사전 준비물
- Node v22+
- `/mnt/c/dev/notes/live-tutorial/`에서 `npm install` 1회 실행 (이미 됨)

### 명령어 / 입력
```bash
# 터미널 1 — watcher 시작 (시연 시작 전 띄워두면 됨)
cd /mnt/c/dev/notes/live-tutorial
npm run watch                      # = node render.mjs

# 브라우저 — HTML 열기 (1회만)
explorer.exe /mnt/c/dev/notes/live-tutorial/tutorial.html
```

### 예상 결과
- watcher 콘솔: `[14:58:15] rendered 578B → tutorial.html` 형태로 매 변경마다 로그
- 브라우저: 1초마다 `<meta http-equiv="refresh">`로 자동 새로고침 → 저장 → 1초 안에 화면 반영
- 우하단에 갱신 시각 스탬프(`14시 58분 36초 갱신`) 떠있어서 청중도 갱신 발생을 인지

### 막히면
| 증상 | 원인 | 해결 |
|---|---|---|
| HTML이 안 만들어짐 | marked 미설치 | `cd tools/live-tutorial && npm install` |
| 변경 후 새로고침 안 됨 | meta refresh가 막혔거나 캐시 | 시크릿 창 / 강제 새로고침 |
| 1초 새로고침이 거슬림 | meta refresh 본래 한계 | 추후 SSE(EventSource)로 업그레이드 가능 (지금은 zero-dep 우선) |
| 한글 깨짐 | charset 설정 누락 | template `<meta charset="UTF-8">` 확인 |

### 청중 시연 멘트 흐름
1. 터미널·브라우저 듀얼 화면 띄움 → "왼쪽이 클로드, 오른쪽이 그 결과 교재예요."
2. 클로드에게 "여기에 H2로 '복습 정리' 추가해줘" → MD 변경 → 1초 내 우측 갱신
3. "이 두 개는 같은 시간선에서 동작합니다. 이게 **클로드 + 단순한 파이프라인 = 라이브 교재**예요."
4. (옵션) `render.mjs` 30줄짜리 코드를 청중에 한 번 펼쳐서 "도구 자체도 단순합니다" 보여줌

### 업그레이드 여지 (시연 후 정리할 거)
- meta refresh → SSE로 교체 (instant reload, 깜빡임 0)
- 코드 블록에 syntax highlight (highlight.js or shiki)
- 다크/라이트 토글
- 단일 페이지 → 좌측 ToC 자동 생성

이번 시연용으론 **현재 30줄짜리 zero-dep 버전이 충분**. 업그레이드는 시연 후.
